= Introducing SrcFlow


  <p>Today I have published a specification for a software change management process called <a href="https://srcflow.com/" rel="external">SrcFlow</a> (pronounced "source flow").</p>

  <p>SrcFlow is a process for managing change in software systems. At its core is a simple, intuitive branching-and-merging strategy for [Git]. SrcFlow also specifies how issues should be managed and prioritized, how changes should be tested and merged, and how software releases should be prepared and versioned.</p>

  <p>This is my preferred change management process for software projects. <a href="https://docs.gitlab.com/ee/topics/gitlab_flow.html" rel="external"></a>GitLab's branching-and-merging strategy</a> provided the initial baseline for my workflow. From there I have iterated my process over several years, testing and refining it in the context of a variety of software projects. I have also drawn inspiration from other documented source control workflows and case studies of how large software houses manage changes to the code and configuration of their software products.</p>
  <h2>Design constraints</h2>
  <p>SrcFlow changed shape over time. But my process settled into a consistent design once I applied a set of constraints to it. Those constraints were:</p>
  <ol>
    <li>Branch naming conventions and integration strategies should <em>feel</em> intuitive and natural. They should closely model our real-world development, testing and release processes.</li>
    <li>Developers should need to remember only a handful of <code>git</code> CLI commands.</li>
    <li>It should be easy to monitor work-in-progress, even in the context of large-scale, high-velocity projects.</li>
    <li>Change requests must be tracked throughout the entire life cycle. All changes to the software system under development – both functional and non-functional changes – should be traceable back to the original business requirements specifications, or the incidents or bug reports, that initiated the changes in the first place.</li>
    <li>The source control workflow must minimize the chances of time-sucking, soul-destroying merge conflicts, even in th context of integrating epic changes.</li>
    <li>There must exist branches of production-grade artifacts that are always in a stable, deployable state.</li>
    <li>The commit history should provide a meaningful changelog, with clearly-signposted release points. Meta data encoded in the source control repository should be in a format that supports automation of recurring procedures such as bumping version numbers.</li>
    <li>The workflow should be scalable to fit projects of all shapes and sizes.</li>
    <li>It should be easy to plug automated systems, including "continuous" integration and delivery services, into the source control pipeline to take care of recurring tasks.</li>
  </ol>
  <p>Really, SrcFlow is a curation of ideas that have emerged as best practices for managing change in software systems. It builds on existing conventions such as [Semantic Versioning] and [Conventional Commits], and finds common ground between various branching and merging strategies, issue tracking strategies, and release strategies.</p>
     No two software houses make software the same way. Every software company seems to have its own unique mix of processes, practices and policies for managing changing requirements in their software products.
     SrcFlow should be considered as a change management _framework_ rather than a rigid _process_. It is intended to be a pragmatic starting point for designing change management procedures for software projects, something that you should tailor for your own specific needs.
    For example, if a project aims to deploy software updates to production systems multiple times every day, and if the release history is always linear (ie if there is no requirement to maintain older major releases), then there may not be much benefit to maintaining a changelog or even  bothering with version numbers. Conversely, high integrity software systems may demand more quality assurance checkpoints than is offered by SrcFlow. SrcFlow can be xxxx and extended as necessary.
        Engineers are encouraged to cherry-pick the parts of SrcFlow that will bring the greatest value to their software projects, and modify or discard the parts that don't fit so well. SrcFlow is like a collection of best practices for source control management that you can tailor to snuggly fit each project's singular operational requirements.
  There could never be a single workflow that fits perfectly every software project. Software varies too widely in scope and complexity. Individual engineering teams need to devise workflows that find the right balance between being simple and frictionless — prioritising velocity of the software's evolution — and being robust and predictable, with enough quality controls to mitigate undesirable regressions in the software under construction — prioritising stability of the evolving software. The optimum balance varies from one project to another.
  SrcFlow provides a framework for that workflow design.
  ------
    Furthermore, these workflows are focused only on the _implementation_ phase of the software engineering lifecycle -- the bit where some computer programming is done. They cover change management at the code and configuration level. But change requests need to be tracked throughout their whole lifecycle. Branching-and-merging strategies need to be integrated with wider engineering operations, including the business process of gathering, analyzing and defining software requirements and user acceptance criteria. Half-baked source control management strategies tend to add friction to engineering processes, dragging on development velocity.
    Without a joined-up approach to software change management, software engineering operations become brittle and volatile. It becomes difficult to monitor work-in-progress, and gaps in quality controls within source control pipelines can allow software defects to slip through to production.

But it is a bit more than that, and encroaches upon the space occupied by software delivery process frameworks, ie wider change management processes. As well as specifying branch naming conventions and integration steps, SrcFlow specifies how issues should be written and tagged, how changes should be reviewed and tested, and how software releases should be prepared and versioned. ... SrcFlow offers a more joined-up approach to software change management. It regulates the whole life cycle of a change request, from initial reporting of an incident or the definition of a newly emerging business requirement, via the design and implementation of a solution, though to testing and integration of the accepted solution in production builds. Branching-and-merging is just one component of SrcFlow.

  <p>Above all, SrcFlow is designed to be scalable to projects of all shapes and sizes. In my opinion, this is <em>the</em> distinguishing feature of SrcFlow. There are many other documented branching-and-merging conventions, but they each suit specific categories of software projects. For example, <a href="https:////nvie.com/posts/a-successful-git-branching-model/" rel="external">GitFlow</a> was fashioned for the intermittent release of version software, and does not scale for continuously delivered, singly versioned software like web apps. <a href="https://about.gitlab.com/2014/09/29/gitlab-flow/" rel="external">GitLab Flow</a> is a better template for such projects. Meanwhile, the minimalist <a href="https://scottchacon.com/2011/08/31/github-flow.html" rel="external">GitHub Flow</a> is a good baseline for community-driven open source projects.</p>
